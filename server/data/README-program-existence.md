# Program-existence pipeline

How the matcher decides "does this school field a varsity soccer program of this gender at this division?" — the single most-flagged source of bad matches early on.

## Why this exists

Early versions of the school matcher trusted the coach scraper's `status === 'no-program'` field as the only negative signal. The scraper produces three flavors of false-positive:

- `'success'` with the wrong-gender coach (Alabama men's resolved to the women's head coach Ashley Johnston).
- `'email-inferred'` with a fabricated coach name (Georgia Tech men's "Albert Demerit", an AI hallucination from the email-format heuristic).
- `'web-name-only'` with a club coach (BYU men's after the 2014 program cut).

Users kept reporting football schools without men's soccer (Alabama, Tennessee, Florida, Georgia, Vanderbilt, Auburn, Texas, Texas A&M, etc.) appearing in men's match results. The fix was a layered pipeline that triangulates Wikipedia + AI verification + a manual override.

## Resolution priority

`hasProgramOfGender(schoolId, gender)` in `server/lib/schoolMatcher.ts` consults sources in this order. First non-undefined answer wins:

1. **Manual override** (`noProgramOverrides.json`) — last word for cases the automated pipeline still gets wrong. Today: 7 Big 12 / SEC men's-no-program entries whose schoolIds don't match anything in EADA / Wikipedia / scraper data (orphan IDs). Audit redundancy with `npx tsx server/scripts/auditOverrides.ts` before adding new entries.

2. **EADA federal data** (`eadaPrograms.json`) — the federally-mandated Equity in Athletics Disclosure Act dataset published annually by the U.S. Department of Education. Every Title IV institution with an athletics program self-reports each year, including varsity participation counts by sport and gender. Covers all divisions (D1 / D2 / D3 / NAIA / NJCAA). Trusted above every other automated signal because it's the federal source of truth. `true` = the school reported PARTIC > 0 for soccer of this gender. `false` = the school filed an EADA report but listed zero soccer participation (or no soccer row at all). `undefined` = no EADA data on file.

3. **AI verification** (`programVerifications.json`, high or medium confidence) — Claude Haiku verdict for cases where Wikipedia and the scraper disagreed. Mostly redundant now that EADA is in the pipeline, but kept as a backup signal for schools where EADA is missing data.

4. **Wikipedia** (`sponsoredPrograms.json`) — D1+D2 lists scraped from Wikipedia. Mostly redundant now that EADA covers all divisions, but useful for cross-validation and as a backup.

5. **Coach status** (`coachesScraped.json`) — legacy fallback. Only `'no-program'` is treated as a negative signal; anything else is permissive.

## Files in this directory

| File | Purpose | Generated by | Consumed by |
|---|---|---|---|
| `eadaPrograms.json` | **Federal source of truth** — varsity sport sponsorship from the EADA bulk file | `buildEadaPrograms.ts` (auto-fetches latest year from ope.ed.gov) | `schoolMatcher.hasProgramOfGender` |
| `wikipediaPrograms.json` | Raw scraped Wikipedia data — four arrays of school names per division/gender | Manual (WebFetch in a Claude conversation) | `buildSponsoredPrograms.ts` |
| `sponsoredPrograms.json` | Wikipedia-derived sponsorship map | `buildSponsoredPrograms.ts` | `schoolMatcher.hasProgramOfGender` |
| `programDisagreements.json` | Read-only audit of where Wikipedia and the scraper disagree | `findProgramDisagreements.ts` | `verifyDisagreements.ts` |
| `programVerifications.json` | Claude Haiku verdicts for the disagreement set | `verifyDisagreements.ts` | `schoolMatcher.hasProgramOfGender` |
| `noProgramOverrides.json` | Manual no-program escape hatch | Hand-edited | `schoolMatcher.hasProgramOfGender` |

## Running the pipeline manually

```bash
# 1. Refresh EADA federal data (auto-downloads the latest academic year)
npx tsx server/scripts/buildEadaPrograms.ts

# 2. Regenerate Wikipedia → schoolId map (uses cached wikipediaPrograms.json)
npx tsx server/scripts/buildSponsoredPrograms.ts

# 3. Find disagreements between Wikipedia and the coach scraper
npx tsx server/scripts/findProgramDisagreements.ts

# 4. Send those disagreements to Claude Haiku for binary verification (~$0.50)
npx tsx server/scripts/verifyDisagreements.ts

# 5. (Optional) Audit which override entries are redundant
npx tsx server/scripts/auditOverrides.ts
```

The monthly GitHub Actions cron (`.github/workflows/refresh-program-existence.yml`) runs steps 1–4 the day after the coach scraper refreshes its data, opening a PR with any new diffs.

## When to refresh `wikipediaPrograms.json`

Wikipedia's per-sport program lists change rarely — typically only when:
- A school adds or drops a varsity program (a few times per year nationally).
- NCAA realignment moves schools between D1 / D2.
- Wikipedia editors fix coverage gaps.

To re-fetch:
1. In a Claude Code conversation, ask Claude to use WebFetch on the four pages:
   - `https://en.wikipedia.org/wiki/List_of_NCAA_Division_I_men%27s_soccer_programs`
   - `https://en.wikipedia.org/wiki/List_of_NCAA_Division_I_women%27s_soccer_programs`
   - `https://en.wikipedia.org/wiki/List_of_NCAA_Division_II_men%27s_soccer_programs`
   - `https://en.wikipedia.org/wiki/List_of_NCAA_Division_II_women%27s_soccer_programs`
2. Update the four arrays in `wikipediaPrograms.json`.
3. Re-run `buildSponsoredPrograms.ts` and `verifyDisagreements.ts`.
4. Commit the updated `sponsoredPrograms.json` + `programVerifications.json`.

Wikipedia per-sport lists for D3 / NAIA / NJCAA do not exist on Wikipedia — those divisions still rely on the scraper + override path. If user complaints about D3/NAIA/JUCO false positives become common, escalate to **Phase 4**: cross-reference with the Equity in Athletics Disclosure Act (EADA) federal database at `https://ope.ed.gov/athletics`.

## When to add a manual override

Add an entry to `noProgramOverrides.json` only when:
- A user reports a wrong school appearing in matches, AND
- Both Wikipedia and AI verification got it wrong (rare — usually one of them is right).

Always add a `_reason_<key>` annotation so future maintainers know why it's there. Run `auditOverrides.ts` before merging — it'll tell you whether the entry is redundant (already handled by Wikipedia/AI) or load-bearing.
